# 관리자 페이지 화면설계안 (BE 기준)

> 목적: 관리자가 **FE 입력(폼데이터) / 이미지 / OCR 결과 / 판정 / 콜백 송출 여부**를 한 화면에서 확인·조치(검수/보정/콜백)할 수 있게 한다.  
> 기준: FastAPI(Admin API) 현재 구현 상태 및 `관리자_페이지_추가_개발_지침.md` 반영 항목.

---

## 1. 정보구조(IA) / 탭

- **판정 규칙**: 운영 파라미터 설정 (`/admin/rules/judgment`)
- **후보 상점**: 신규 상점 후보 검수 (`/admin/stores/candidates`, approve)
- **신청 검색**: submission 검색/상세/검수/콜백 (`/admin/submissions*`)

---

## 2. 공통 UI 요소

### 2.1 인증 입력(헤더)

- 입력: `Admin Key(선택)`, `담당자(actor)`
- 동작:
  - `ADMIN_API_KEY`가 설정된 환경에서는 `X-Admin-Key` 필요
  - 감사로그에는 `X-Admin-Actor`가 `actor`로 저장됨(콜백/검수 추적에 사용)

### 2.2 상태 뱃지 규칙(권장)

- **진행 중**: `PROCESSING`, `VERIFYING`
- **수동 검토 필요**: `PENDING_NEW`, `PENDING_VERIFICATION`
- **종결**: `FIT`, `UNFIT_*`, `ERROR`, `ERROR_OCR` 등 (또는 `shouldPoll=false`)

---

## 3. 화면: 신청 검색(목록)

### 3.1 목적

- 특정 `receiptId` 추적
- 수동 검토 대상(`PENDING_*`)만 빠르게 추려 검수 진행
- 장기 처리 중(`PROCESSING/VERIFYING`) 감시

### 3.2 검색/필터

- 입력:
  - `receiptId`
  - `status` (전체/FIT/UNFIT/PENDING_NEW/PENDING_VERIFICATION/VERIFYING/PROCESSING 권장)
  - `dateFrom`, `dateTo`

### 3.3 목록 테이블 컬럼(권장)

- `receiptId`(클릭 시 상세)
- `project_type`(STAY/TOUR)
- `status`
- `total_amount`
- `created_at`
- (선택) `global_fail_reason` 요약

### 3.4 목록 내 빠른 액션(선택)

- “상세”만 제공해도 충분

---

## 4. 화면: 신청 상세(핵심)

### 4.1 레이아웃(권장: 3열)

**좌(증빙 이미지)** / **중(OCR 결과)** / **우(FE 입력 + 운영 액션)**

---

### 4.2 좌: 증빙 이미지 패널

- API: `GET /api/v1/admin/receipts/{receiptId}/images`
- UI:
  - 장별 썸네일 리스트 + 클릭 확대(모달/새 탭)
  - `doc_type` 표시(예: `RECEIPT`, `OTA_INVOICE`)
- 기대 효과:
  - “업로드는 되었는데 깨진 파일”/“영수증이 아닌 이미지” 등을 즉시 육안 확인

---

### 4.3 중: OCR 결과 패널

#### 4.3.1 장별 요약 테이블(권장)

각 장(ReceiptItem)별로:
- `seq_no`, `doc_type`
- `status`
- `error_code`, `error_message`
- `confidence_score`
- `store_name`, `biz_num`, `address`
- `pay_date`, `amount`, `card_num`

#### 4.3.2 카드번호 표기 규칙

- `0000` → “현금”
- `1000` → “카드번호 없음/****”
- 그 외 4자리 → “****-****-****-1234”

#### 4.3.3 OCR 원문(선택)

- `ocr_raw`는 크므로 “원문 보기” 토글로 접힘 제공(기본 숨김 권장)

---

### 4.4 우: FE 입력(폼데이터) + 비교 + 운영 액션

#### 4.4.1 FE 입력 스냅샷

- 소스: `submissions.user_input_snapshot`
- 목적:
  - “FE 입력이 제대로 왔는지”
  - “OCR 결과와 비교해서 수동 검수 판단”

#### 4.4.2 비교 UI(권장)

- 장 단위 비교(방식2):
  - 좌표: `documents[i]`(이미지) ↔ `data.items[i]`(FE 입력) ↔ `receipt_items[i]`(OCR 결과)
- 표시 우선순위:
  - `amount`, `pay_date`, `location`(검증에 직접 영향)
  - `store_name`, `biz_num`, `address`(인식 불량/정합 검증)

#### 4.4.3 운영 액션 버튼(필수)

- **Override(검수 완료)**  
  - API: `POST /api/v1/admin/submissions/{receiptId}/override`
  - 입력: `status(FIT/UNFIT)`, `reason`, `resend_callback`
  - 운영 권장: **“검수 완료 시 콜백 자동 송출(resend_callback)” 기본 ON**

- **콜백 검증(즉시 송출)**  
  - API: `POST /api/v1/admin/submissions/{receiptId}/callback/verify`
  - 출력: `{ ok, status, elapsed_ms, url }`를 UI에 즉시 표시

- **콜백 로그 조회**  
  - API: `GET /api/v1/admin/submissions/{receiptId}/callback/logs?limit=...`
  - UI: 최근 10~20개를 타임라인으로 노출
  - 해석:
    - `CALLBACK_SEND.meta.ok=true` + `meta.url=콜백URL` → 송출 성공
    - `CALLBACK_VERIFY` → “검증 버튼 클릭 이벤트”
    - `CALLBACK_RESEND` → “재전송 트리거 기록”

---

## 5. 운영 시나리오별 권장 UX

### 5.1 AUTO_PROCESSING(진행 중)

- 상태: `PROCESSING/VERIFYING`, `shouldPoll=true`, `statusStage=AUTO_PROCESSING`
- UI:
  - “분석 중” 뱃지
  - 콜백 검증 버튼은 **가능**(필요 시), 단 결과가 최종이 아닐 수 있음 안내

### 5.2 MANUAL_REVIEW(수동 검토)

- 상태: `PENDING_NEW` 또는 `PENDING_VERIFICATION` (`reviewRequired=true`)
- UI:
  - “수동 검토 필요” 뱃지 + 사유(`error_code`) 강조
  - FE 입력/이미지/OCR을 나란히 보도록 기본 펼침
  - 검수 완료(Override) 버튼 강조

### 5.3 DONE(종결)

- 상태: `shouldPoll=false`
- UI:
  - 최종 판정/사유 고정 표시
  - 필요 시 콜백 재전송/검증만 제공

---

## 6. BE 연동 포인트(요약)

- 상세 데이터: `GET /api/v1/admin/submissions/{receiptId}`
  - `user_input_snapshot` 포함
  - status payload에 `items[]`(장별 OCR 결과, error_code, confidence 등) 포함
- 이미지: `GET /api/v1/admin/receipts/{receiptId}/images`
- 검수 완료: `POST /api/v1/admin/submissions/{receiptId}/override`
- 콜백:
  - 검증: `POST /api/v1/admin/submissions/{receiptId}/callback/verify`
  - 로그: `GET /api/v1/admin/submissions/{receiptId}/callback/logs`
  - 재전송: `POST /api/v1/admin/submissions/{receiptId}/callback/resend`

---

## 7. 운영자가 “꼭 확인해야 하는 4가지” (화면에 고정 권장)

1. **FE 입력(user_input_snapshot)이 존재하는지** / 장별 입력이 맞는지  
2. **이미지가 정상 렌더링되는지**(깨짐/JSON/multipart 업로드 의심)  
3. **OCR 핵심필드(store/biz/address)가 비어있는지 + confidence 낮은지** (`OCR_004` 확인)  
4. **콜백이 실제 송출되었는지** (`CALLBACK_SEND.meta.ok/url/status`)
