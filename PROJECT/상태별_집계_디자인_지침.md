# GEMS 상태별 집계(Status Aggregation) 디자인 지침

지자체 보안 지침과 **AI 학습 자산화**라는 GEMS의 핵심 목표를 시각적으로 구현하기 위한 **상태별 집계 화면** UI/UX 디자인 지침입니다.  
단순 테이블 형태를 넘어, 캠페인 정책(최소 금액 6만/5만 원) 달성 여부와 **데이터 교정 자산 가치**를 한눈에 파악할 수 있도록 고도화하는 가이드입니다.

---

## 1. 시각적 계층화 및 컬러 시스템 (Visual Hierarchy)

데이터의 성격에 따라 명확한 컬러 코딩을 적용하여 관리자의 즉각적인 판단을 돕습니다.

| 구분 | 컬러 | 의미 |
|------|------|------|
| **적합 (FIT)** | Green | 캠페인 정책을 통과한 **청정 데이터**이자 AI 학습용 **골든 데이터셋** |
| **부적합 (UNFIT)** | Crimson/Red | 정책 미달(금액, 지역 등) 건. 감사 시 **부정수급 방지 성과**로 강조 |
| **대기/검증 (PENDING)** | Amber/Blue | 검수 프로세스가 진행 중인 **운영 유동 데이터** |
| **에러 (ERROR)** | Destructive/Red | OCR 오류·멈춤 등. **재처리·모니터링** 대상 |

**FE 적용 예**
- 적합: `bg-green-500/10`, `text-green-800` (dark: `dark:bg-green-600/15`, `dark:text-green-200`)
- 부적합: `bg-red-500/10`, `text-red-800` (dark 대응)
- 대기 및 검증: `bg-amber-500/10`, `text-amber-800`
- 에러: `bg-destructive/10`, `text-destructive`

---

## 2. 캠페인 정책 연동형 지표 (Policy-Driven Metrics)

단순 건수 합산이 아닌, **판정 규칙(환경설정)** 에서 설정된 정책 기반의 심층 분석을 제공합니다.

| 지표 | 설명 | 활용 목적 |
|------|------|-----------|
| **금액 미달 세분화** | 부적합 건 중 **합산 금액 미달(UNFIT_AMOUNT)** 건을 별도 표기 | 담당자가 영수증을 보고 금액을 교정할 수 있는 **교정 기회** 식별 |
| **최소 금액 기준** | 숙박(STAY) 6만 원, 관광(TOUR) 5만 원 등 캠페인별 최소 금액 | 정책 달성 여부·비교 표시 |
| **지역 외 필터링** | 캠페인 대상 지역 외에서 발생한 영수증(UNFIT_REGION) 건을 별도 섹션으로 분리 | **캠페인 타겟팅 정확도** 측정·감사 |

**BE/API 연동**
- 부적합 사유별 건수: `UNFIT_AMOUNT`, `OUT_OF_PERIOD`, `UNFIT_REGION`, `DUPLICATE` 등 상태 코드별 집계 제공 시 FE에서 세분화 표시 가능.
- 캠페인별 `min_amount_stay`, `min_amount_tour` 연동 시 "정책 대비 달성률" 등 정책 연동형 KPI 표시 가능.

---

## 3. AI 자산화 및 교정 데이터 KPI (Assetization KPI)

담당자의 수정 작업이 시스템에 미친 기여도를 수치화하여 **데이터 자산**으로서의 가치를 증명합니다.

| KPI | 설명 | 시각화 제안 |
|-----|------|-------------|
| **데이터 교정률 (Human Correction Rate)** | OCR이 UNFIT으로 판정했으나, 담당자가 금액 등을 교정하여 **FIT으로 전환한 건수** | "교정 자산" 카드·배지 또는 별도 집계 행 |
| **학습용 데이터 확보량** | FIT 상태 중 **이미지 + Sidecar JSON**이 완벽히 매핑된 학습용 데이터셋 구축 현황 | **프로그레스 바** 또는 목표 대비 건수 |

**BE/API 연동**
- 교정 이력(Sidecar·human_correction) 존재 여부 또는 `UNFIT → FIT` 전환 건수 집계 API 제공 시 FE에서 "교정 자산" 수치 표시 가능.
- 학습용 데이터셋 조건(이미지 + Sidecar 완비) 충족 건수 API 제공 시 프로그레스 바 연동 가능.

## 4. 개선된 집계 레이아웃 구성 (Proposed Layout)

| 구분 | 핵심 위젯·시각화 요소 | 활용 목적 |
|------|------------------------|-----------|
| **Status Cards** | 적합/부적합/대기 건수 + **(선택) 전일 대비 증감율** | 실시간 운영 흐름 파악 |
| **분류별 아코디언** | 적합 \| 부적합 \| 대기 및 검증 \| 에러 — 클릭 시 상세 내역 | 계층적 정보 노출·공간 절약 |
| **Policy Chart** | 숙박(STAY) vs 관광(TOUR) **정책 달성 비중** (또는 업종별 건수 비중) | 캠페인 유형별 참여도·정책 달성 분석 |
| **Reason Analysis** | 부적합 사유별 **도넛/파이 차트** (금액 미달, 지역 외, 기간 외 등) | 운영 정책 개선 근거 마련 |
| **AI Asset Feed** | **(선택)** 금일 확보된 **골든 데이터셋** 건수 및 교정 건수 | AI 모델 고도화 성과 측정 |

**현재 구현**
- 분류별 아코디언(적합 \| 부적합 \| 대기 및 검증 \| 에러), 분류별 합계 + 클릭 시 상세 테이블.
- 컬러 시스템 적용, 엑셀(CSV) 다운로드, 10분 캐시·집계 새로고침.

**추가 구현 권장**
- Status Cards(전일 대비 증감율), Policy Chart(STAY/TOUR 비중), Reason Analysis(부적합 사유 도넛 차트), AI Asset Feed(교정률·학습용 데이터 프로그레스).

## 5. 데이터 정합성 및 보안 지침 (GEMS Standard)

### 5.1 실시간 집계 vs 캐시

| 항목 | 권장 사항 |
|------|-----------|
| **캐시** | 대량 영수증(수천 건) 처리 시 성능을 위해 **10분 단위 캐시** 유지 (FE `staleTime: 10 * 60 * 1000`) |
| **즉시 집계** | 중요한 감사·보고 시점에는 **"집계 새로고침"** 또는 **(선택) "즉시 집계"** 버튼으로 최신 데이터 재조회 |
| **마지막 집계 시각** | "마지막 집계: MM/dd HH:mm" 표시로 데이터 시점 투명성 확보 |

### 5.2 엑셀 다운로드 보안

| 항목 | 권장 사항 |
|------|-----------|
| **비식별화** | 집계 내역 다운로드 시, 개인정보 보호를 위해 **비식별화된 데이터**(건수·상태·업종 등 집계치)만 포함 |
| **전체 데이터** | 감사 목적의 개인 식별 정보가 포함된 전체 데이터 다운로드는 **별도 승인 절차**를 거치도록 정책 수립 |
| **다운로드 형식** | CSV/Excel에 포함되는 컬럼을 **구분·STAY·TOUR·합계** 등 집계 수준으로 제한하고, receiptId·userUuid 등은 제외 권장 |

### 5.3 Sidecar·Hash 연동

| 항목 | 권장 사항 |
|------|-----------|
| **Sidecar 동기화** | 모든 집계 수치는 보안 스토리지 내 개별 **Sidecar JSON** 값과 실시간(또는 주기 동기화)으로 일치하도록 BE 설계 |
| **Hash 검증** | 데이터 위변조 방지를 위해 **Hash 검증** 결과를 주기적으로 리포팅하고, 필요 시 관리자 화면에 "검증 완료" 등 상태 표시 |

## 6. 분류·용어 통일 (서브미션 상태와 동일)

상태별 집계 화면의 **분류명·상태 라벨**은 서브미션(제출) 목록·검수 큐 등과 동일한 용어를 사용합니다.

| 분류 | 포함 상태 코드 | 표시 라벨 (예) |
|------|----------------|-----------------|
| **적합** | FIT, APPROVED, DONE | 적합, 승인, 완료 |
| **부적합** | UNFIT_AMOUNT, OUT_OF_PERIOD, UNFIT_REGION, DUPLICATE, UNFIT, REJECTED | 금액 미달, 기간 외, 지역 외, 중복 의심, 부적합, 반려 |
| **대기 및 검증** | PENDING_NEW, MANUAL_REVIEW, PROCESSING, VERIFYING, PENDING_VERIFICATION, PENDING | 신규 대기, 검토 대기, 처리 중, 검증 중, 검증 대기, 대기 |
| **에러** | ERROR_OCR, STUCK, AMOUNT_MISMATCH, DUPLICATE_SUSPECT + 기타 미분류 | OCR 오류, 멈춤, 금액 불일치, 중복 의심 + 코드 또는 "기타" |

FE는 `getStatusLabelKo(status)`(또는 `STATUS_LABEL_KO`)를 사용하여 전 화면에서 동일 라벨을 적용합니다.
