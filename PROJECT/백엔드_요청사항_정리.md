# GEMS Admin 백엔드(BE) 요청 사항 정리

관리자 페이지(프론트엔드)에서 필요로 하는 API 및 데이터 요청 사항을 정리한 문서입니다.

---

## 1. 로그인 API – 조직 정보 포함

**상세:** [로그인_API_조직정보_요청사항.md](./로그인_API_조직정보_요청사항.md)

- **POST /api/v1/auth/login** 응답의 `user` 객체에 아래 필드 포함 요청.
  - `organization_id` / `organizationId`: 조직 ID (미반환 시 프론트 "api (미반환)" 표시)
  - `org_name` / `orgName`: (선택) 조직명
  - `org_type` / `orgType`: (선택) 조직 유형
  - `campaignIds`: (선택) 해당 사용자가 접근 가능한 캠페인 ID 배열 → 기관/검수자 권한 필터에 사용

---

## 2. 제출(Submissions) API – 대시보드·검수용

현재 **GET /api/v1/admin/submissions** 를 전제로 한 추가/확인 사항입니다.

### 2.0 422 Unprocessable Content 대응 (우선 요청)

아래와 같이 호출 시 **422**가 발생하고 있습니다.  
→ **원인**: 쿼리 파라미터(`status`, `from`, `to`, `campaignId`)를 BE에서 아직 허용하지 않거나, 파라미터 이름이 다를 가능성.

| FE가 보내는 예시 | 용도 |
|------------------|------|
| `?status=APPROVED&campaignId=1&limit=1000&offset=0` | 승인 건 금액 합계(예산 소진률) |
| `?from=2026-03-04&to=2026-03-10&campaignId=1&limit=500` | 최근 7일 접수 추이·유형별 비중 |

**BE 측 요청 사항**

- **옵션 A**: 위 파라미터 이름 그대로 지원  
  - `status`, `from`, `to`, `campaignId` (camelCase), `limit`, `offset` 수용
- **옵션 B**: snake_case만 지원하는 경우  
  - 지원하는 파라미터 이름을 알려주시면 FE에서 맞춰서 요청하도록 수정 가능  
  - 예: `campaign_id`, `date_from`, `date_to` 등

422가 나오는 동안은 FE에서 해당 요청을 빈 데이터(`total: 0`, `items: []`)로 처리해 두었으며, BE 반영 후에는 자동으로 실제 데이터가 표시됩니다.

---

### 2.1 쿼리 파라미터 지원

| 파라미터 | 용도 | 비고 |
|----------|------|------|
| `from` | 시작일 (YYYY-MM-DD) | 금일/어제/최근 7일 집계에 사용 |
| `to` | 종료일 (YYYY-MM-DD) | |
| `campaignId` | 캠페인 ID로 필터 | 기관·검수자: 자신의 캠페인만 조회 |
| `status` | MANUAL_REVIEW, APPROVED 등 | 검수 대기 건수, 승인 건 집계에 사용 |
| `limit`, `offset` | 페이징 | |

- **응답**: `{ total: number, items: AdminSubmissionSummary[] }` 형태 유지 요청.
- **items[]** 각 항목에 **project_type** (또는 projectType) 필드 있으면 좋음: `STAY`(숙박) / `TOUR`(일반 소비) 등. 대시보드·서브미션 업종 필터에 사용.

**제출 항목 확장 필드 (선택, FE에서 미제공 시 비활성 표시)**

| 필드 | 용도 |
|------|------|
| `thumbnail_url` | 서브미션 목록·호버 시 썸네일 미리보기 |
| `confidence` | 신뢰도 슬라이더 필터 (0~100, 낮은 건만 검수 우선) |
| `integrityCheck` (boolean) | 무결성 검사 결과, 목록에 OK 표시 |

### 2.2 (선택) 대시보드 전용 집계 API

한 번의 호출로 대시보드 수치를 내려받고 싶을 경우 아래 형태의 API를 검토해 주시면 좋습니다.

- **GET /api/v1/admin/dashboard/stats** (또는 유사 경로)
- **쿼리**: `campaignId`, `from`, `to` (예: 오늘, 어제, 최근 7일)
- **응답 예시**:
  - `todayCount`, `yesterdayCount`: 금일/전일 제출 건수 (전일 대비 증감률 계산용)
  - `pendingCount`: MANUAL_REVIEW 건수
  - `approvedAmountSum`: 승인(APPROVED) 건의 total_amount 합계 (예산 소진률용)
  - `byCategory`: `{ STAY: number, TOUR: number }` 등 (유형별 건수)
  - `dailyCounts`: `{ date: string, count: number }[]` (일자별 추이 차트용)

미구현 시에도 현재처럼 **GET /api/v1/admin/submissions** 여러 번 호출로 프론트에서 집계 가능하나, 성능·일관성 측면에서 집계 API가 있으면 유리합니다.

## 3. 일괄 반려 API (Bulk Rejection)

서브미션(제출 검색) 페이지에서 **일괄 반려 사유 입력 팝업**으로 선택 건에 대해 반려 처리·알림·학습 데이터 태깅을 할 수 있도록, 아래 API 및 연동을 요청합니다.

### 3.1 일괄 반려 실행 API

- **POST /api/v1/admin/submissions/bulk-reject** (또는 유사 경로)
- **Request Body 예시**:
  ```json
  {
    "receiptIds": ["uuid-1", "uuid-2", ...],
    "reasonCode": "image_unreadable",
    "reasonMessage": "제출하신 영수증의 글자가 흐려 금액 확인이 어렵습니다. 재촬영 후 제출 바랍니다.",
    "tagAsError": true
  }
  ```
  - `reasonCode`: 프리셋 ID (예: image_unreadable, duplicate, out_of_scope, below_min_amount)
  - `reasonMessage`: 시민에게 전달할 반려 사유 문구 (알림톡 등에 사용)
  - `tagAsError`: true 시 Sidecar JSON 등에 `is_error: true` 학습 데이터 태그

- **기대 동작**:
  - 각 submission의 `status`를 `UNFIT` 또는 `REJECTED`로 변경
  - (선택) 각 userUuid에 매핑된 연락처로 반려 사유 포함 카카오 알림톡/푸시 발송
  - (선택) 보안 스토리지 Sidecar JSON에 `rejection_reason`, 검수자 ID, 일시 기록
  - `tagAsError: true`인 경우 해당 건을 부정수급/오류 학습 데이터셋으로 분류

### 3.2 감사(Audit) 로그

- 일괄 반려 실행 시 **누가, 언제, 몇 건, 어떤 사유로** 실행했는지 Audit Log에 기록 요청 (IP 포함 시 지자체 보안 감사 대비).

---

## 4. 반려 사유 집계 (선택)

대시보드 “주요 반려 사유 Top 3” 표시를 위해, 반려 사유별 건수를 내려주는 API가 있으면 좋습니다.

- 예: **GET /api/v1/admin/receipts/reject-reasons** (또는 기존 제출/영수증 API 확장)
- **응답 예시**: `[{ reason: "이미지 흐림", count: 32 }, { reason: "금액 불일치", count: 15 }, ...]`
- 미구현 시 프론트에서는 “API 연동 시 표시” 문구만 노출합니다.

---

## 5. 운영 컨텍스트 (Admin Context)

- **GET /api/v1/admin/context**: 현재 관리자의 선택 캠페인(projectId) 등 저장값 조회
- **PUT /api/v1/admin/context**: 선택 캠페인 등 저장  
→ 이미 008_admin_context 마이그레이션 및 라우트 구현된 경우, 유지해 주시면 됩니다.

---

## 6. 캠페인(Campaigns) API

- **GET /api/v1/admin/campaigns**, **GET /api/v1/admin/campaigns/:id**  
  - 응답에 **start_date**, **end_date**, **budget** 포함되어 있으면 대시보드 “캠페인 기간(D-Day)”, “예산 소진 현황”에 사용합니다.
- **campaignId** 기준 제출/집계 API와 연동되면, 기관·검수자는 자신에게 허용된 캠페인만 조회 가능해야 합니다.

---

## 7. 영수증 이미지 (보안 스토리지)

- **GET /api/v1/admin/receipts/:receiptId/images**  
  - 보안 스토리지(Secure Storage) 사용 시, 원본 이미지는 **임시 Signed URL** 등으로 단기간만 노출되도록 구성해 주시면, 지자체 보안 가이드라인 준수에 유리합니다.

---

## 8. 영수증 상태 조회 – 열람 이력·데이터 자산화

영수증 상태 조회 화면에서 **증거 보기(모달)** 사용 시 다음 연동이 있으면 좋습니다.

- **열람 이력 기록**: 증거 이미지/상세를 열람할 때마다 **관리자 ID**, **조회 시간**, **조회 IP**, **receiptId**를 감사 로그에 기록해 주시면, 개인정보 보호법·지자체 보안 지침 대응에 유리합니다.
- **데이터 자산화 태깅 (선택)**: 조회 화면에서 “AI 학습용 우수 사례” / “부정수급 의심 사례”로 분류하는 API가 있으면, 해당 receiptId를 학습 데이터셋 폴더 또는 태그와 연동해 줄 수 있습니다.
  - 예: **POST /api/v1/admin/receipts/:receiptId/tag** Body `{ "tag": "excellent_sample" | "suspected_fraud" }` 등

## 요약 체크리스트

| 항목 | 우선순위 | 비고 |
|------|----------|------|
| 로그인 응답에 organization_id, org_name, org_type, campaignIds | 높음 | [로그인_API_조직정보_요청사항.md](./로그인_API_조직정보_요청사항.md) 참고 |
| GET /admin/submissions 에 from, to, campaignId, status 지원 | 높음 | 422 해결 후 대시보드·검수 필터 |
| 제출 항목에 project_type (STAY/TOUR) 포함 | 중간 | 대시보드·서브미션 업종 필터 |
| 제출 항목 확장: thumbnail_url, confidence, integrityCheck | 선택 | 서브미션 미리보기·신뢰도·무결성 |
| POST 일괄 반려 API (bulk-reject) | 중간 | 서브미션 일괄 반려 팝업 연동 |
| 일괄 반려 시 Audit Log (실행자, IP, 건수, 사유) | 권장 | 지자체 보안·감사 대비 |
| 대시보드 전용 집계 API (stats) | 선택 | 성능·일관성 |
| 반려 사유 집계 API | 선택 | “주요 반려 사유 Top 3” |
| 캠페인 start_date, end_date, budget 노출 | 높음 | 이미 있으면 유지 |
| 영수증 이미지 Signed URL 등 보안 처리 | 권장 | 보안 가이드라인 |
| 영수증 증거 열람 시 감사 로그(관리자 ID, 시각, IP) | 권장 | 상태 조회 화면 증거 보기 |
| 데이터 자산화 태깅 API (우수/부정수급 사례) | 선택 | 상태 조회 화면 태깅 버튼 |

이 문서는 프론트엔드 구현 기준으로 작성되었으며, BE 스펙이 다를 경우 연동 시 조정이 필요할 수 있습니다.
