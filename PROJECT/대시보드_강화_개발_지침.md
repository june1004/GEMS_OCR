# 대시보드 강화 개발 지침 (Dashboard Enhancement Guideline)

> 현재 **전체 제출 데이터**를 보여주는 초기 대시보드를 바탕으로, **숙박/소비 구분** 및 **시계열(월/일/시간) 분석** 기능을 강화하기 위한 개발 지침입니다.  
> **GEMS 보안 스토리지**에 저장된 **Sidecar JSON** 데이터를 실시간 집계하여, AI 학습 자산으로서의 가치를 대시보드에 직접 투영하는 데 초점을 맞춥니다.

## 1. 전체 현황(Overall Status) 보강

영수증 성격에 따른 예산 집행 현황과 데이터 품질을 한눈에 파악합니다.

### 1.1 숙박(STAY) vs 소비(TOUR) 비중 차트

| 항목 | 내용 |
|------|------|
| **시각화** | 전체 제출물 중 숙박·소비 비율을 **도넛 차트(Doughnut)** 로 표시 |
| **데이터 소스** | DB `project_type` 필드 및 Sidecar JSON의 업종 분류 데이터 |
| **FE 참고** | 현재 `Dashboard.tsx`의 `STAY_COLOR` / `TOUR_COLOR` 및 `getAdminSubmissionsBreakdown` 연동 유지·확장 |
| **BE 요청** | `GET /api/v1/admin/submissions` 또는 전용 집계 API에서 `project_type`(STAY/TOUR)별 건수·비율 제공 |

### 1.2 정책 달성 현황 위젯

| 항목 | 내용 |
|------|------|
| **내용** | 설정된 최소 금액(**숙박 60,000원 / 소비 50,000원**)을 넘긴 **적합(FIT)** 데이터의 비중 표시 |
| **목표** | 단순 수집을 넘어 실제 예산이 지원될 **유효 자산** 규모 파악 |
| **지표 예** | "정책 충족 FIT: N건 (전체 대비 O%)", "유효 예산 자산: N건" |
| **BE 요청** | FIT 건수 + (선택) 금액 기준 충족 건수 집계. 캠페인별 `min_amount_stay` / `min_amount_tour` 반영 시 정책 대비 달성률 계산 가능 |

## 2. 시계열 분석 현황표 (Temporal Analysis)

데이터 유입 패턴을 분석하여 **행정 인력 배치** 및 **캠페인 효과** 측정에 활용합니다.

| 구분 | 시각화 방식 | 개발 핵심 내용 |
|------|-------------|----------------|
| **월별 (Monthly)** | **막대 차트 (Bar)** | 캠페인 기간(예: 2026-01-01 ~ 10-31) 내 **월별 유입량** 및 **예산 소진 속도** 비교 |
| **일별 (Daily)** | **영역 차트 (Area)** | 주말/평일 유입 차이 분석. 특정 축제·이벤트 시점의 **피크(Peak)** 탐지 |
| **시간대별 (Hourly)** | **히트맵 (Heatmap)** | 00~23시 중 접수가 집중되는 시간 파악. **검수 담당자 업무 집중 시간대** 가이드 제공 |

### 2.1 월별 (Monthly)

- **차트**: Bar Chart (Recharts `BarChart`, `Bar`, `XAxis`, `YAxis` 등)
- **축**: X = 월(YYYY-MM 또는 "N월"), Y = 제출 건수 (선택: 누적 예산 소진량)
- **데이터**: `created_at` 또는 Sidecar 메타데이터 기준 월별 집계
- **BE 요청**: `from`/`to`로 캠페인 기간 지정 시 **월별 건수** 집계 API 또는 기존 제출 API의 날짜 필터 + FE 집계

### 2.2 일별 (Daily)

- **차트**: Area Chart (Recharts `AreaChart`, `Area`)
- **용도**: 주말/평일 패턴, 이벤트 기간 피크 분석
- **데이터**: 일별 제출 건수 (날짜 문자열 YYYY-MM-DD)
- **BE 요청**: 일별 집계 또는 `from`/`to` + `limit`으로 구간 조회 후 FE에서 일별 그룹핑

### 2.3 시간대별 (Hourly)

- **차트**: Heatmap (요일 또는 일자 × 0~23시)
- **용도**: 접수 집중 시간대 표시 → 검수 인력 배치 참고
- **데이터**: `created_at` 시각(또는 Sidecar 메타) 기준 시간대별 건수
- **BE 요청**: 시간대별 집계 API 또는 raw 제출 목록의 `created_at` 시분초 제공 후 FE에서 시간대(0~23) 집계

## 3. 데이터 아키텍처 및 성능 지침 (GEMS 표준)

대량 영수증 데이터를 빠르고 안전하게 처리하기 위한 백엔드·FE 공통 지침입니다.

### 3.1 Sidecar JSON 기반 집계

| 원칙 | 설명 |
|------|------|
| **원본 이미지 미분석** | 통계 화면 로딩 시마다 **원본 이미지를 재분석하지 않음** |
| **메타데이터만 쿼리** | 스토리지에 저장된 **Sidecar JSON**의 메타데이터만 조회하여 속도 극대화 |
| **BE 역할** | Sidecar 또는 DB에 정규화된 `project_type`, `status`, `created_at`, 금액 등만 집계용으로 제공 |

### 3.2 집계 캐싱 전략

| 항목 | 내용 |
|------|------|
| **캐시 정책** | **10분 캐시** 적용 — 동일 쿼리 10분 이내 재사용으로 서버 부하 감소 |
| **실시간 반영** | **[집계 새로고침]** 버튼으로 필요 시 캐시 무시·재조회하여 최신 데이터 반영 |
| **FE 구현** | React Query `staleTime: 10 * 60 * 1000`(10분) + "집계 새로고침" 버튼에서 `queryClient.invalidateQueries` 또는 `refetch` |

### 3.3 보안 격리 조회 (IAM)

| 항목 | 내용 |
|------|------|
| **원칙** | 대시보드 통계 **클릭 시 리스트로 이동**할 때, 해당 담당자의 **소속(지자체/기관)** 에 맞는 데이터만 노출 |
| **적용** | IAM 권한에 따라 조회 가능한 `project_id` / 지역 / 캠페인 범위 제한. BE에서 **권한 필터 강제** |
| **FE** | 동일 세션·토큰으로 제출 검색/리스트 API 호출 시, BE가 이미 필터링한 결과만 수신 |

## 5. 구현 체크리스트 (FE/BE)

### 5.1 FE (대시보드 페이지)

- [ ] **숙박/소비 도넛 차트** — `project_type`(STAY/TOUR) 비율, 기존 Pie/Donut 확장
- [ ] **정책 달성 위젯** — 최소 금액 충족 FIT 비중, 캠페인 정책 연동
- [ ] **월별 막대 차트** — 캠페인 기간 월별 유입량, 클릭 시 일별 드릴다운
- [ ] **일별 영역 차트** — 일별 제출 건수, 주말/피크 강조
- [ ] **시간대별 히트맵** — 0~23시(및 요일) 집중도
- [ ] **집계 새로고침 버튼** — 10분 캐시 무시 및 재조회
- [ ] **AI 학습 데이터셋 OOO건** — 구석 고정 문구
- [ ] **Deep Navy 테마** 유지 및 인터랙션(호버·클릭) 일관성

### 5.2 BE / API

- [x] **project_type별 집계** — **GET /api/v1/admin/dashboard/stats** 응답 **byCategory** (STAY/TOUR 건수). 도넛 차트용.
- [x] **월/일/시간대 집계** — 동일 **dashboard/stats** 에서 **dailyCounts**(일별), **monthlyCounts**(월별), **hourlyCounts**(0~23시). 캠페인·IAM 필터 적용. 상세: [백엔드_요청사항_정리.md](./백엔드_요청사항_정리.md) §2.2.1.
- [ ] **10분 캐시** — BE는 서버 캐시 미적용(매 요청 DB 조회). FE에서 `staleTime: 10 * 60 * 1000` 및 "집계 새로고침" 시 refetch 권장.
- [x] **IAM** — AdminContext 기반: 비수퍼관리자는 **campaign_ids** 로만 목록·집계 조회. 대시보드→리스트 이동 시 동일 권한으로 필터링됨. §2.2.1 참고.

## 6. 참고: 현재 대시보드 구조 (코드 기준)

- **파일**: `src/pages/Dashboard.tsx`
- **주요 쿼리**: `getAdminSubmissionsSafe`, `getAdminSubmissionsBreakdown`, `getAdminContext`, 캠페인·금일/어제/최근 7일 제출 등
- **색상**: `STAY_COLOR`, `TOUR_COLOR` (CSS 변수 `--dashboard-stay`, `--dashboard-tour`)
- **차트**: Recharts `AreaChart`, `PieChart`, `LineChart` 등 이미 사용 중 → Bar, Heatmap 추가 시 동일 라이브러리 활용 권장

이 지침은 **GEMS 보안 스토리지·Sidecar JSON**과 연동된 실시간 집계 및 AI 자산화 지표를 대시보드에 반영하기 위한 공통 기준입니다.

**BE 집계 API (현재 구현)**  
**GET /api/v1/admin/dashboard/stats** — 쿼리: `campaignId`, `from`, `to`, `timezone`.  
응답: `todayCount`, `yesterdayCount`, `pendingCount`, `approvedAmountSum`, **byCategory**(STAY/TOUR), **dailyCounts**, **monthlyCounts**(`[{ month: "YYYY-MM", count }]`), **hourlyCounts**(`[{ hour: 0~23, count }]`), `byStatus`, `minAmountStay`, `minAmountTour`, `lastAggregatedAt`.  
IAM: 비수퍼관리자는 접근 가능한 `campaign_ids` 기준으로만 집계됨.
