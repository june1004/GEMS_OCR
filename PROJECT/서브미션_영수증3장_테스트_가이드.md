# 서브미션(영수증 3장) FastAPI 테스트 가이드 — DB 기준 예시

DB에 있는 `submissions` / `receipt_items` 값을 이용해 **Presigned URL → Complete** 흐름을 테스트하는 방법입니다.

---

## 1. DB에서 테스트용 값 조회

DBeaver 등에서 **실제 사용 중인 DB**에 접속한 뒤 아래 SQL을 실행합니다.

### 1-1. 영수증 3개인 서브미션 1건 찾기

```sql
-- submission_id, user_uuid, 영수증 개수, 각 image_key(object_key) 조회
SELECT
    s.submission_id,
    s.user_uuid,
    s.status,
    s.project_type,
    COUNT(ri.item_id) AS receipt_count,
    array_agg(ri.image_key ORDER BY ri.seq_no, ri.item_id) AS image_keys
FROM submissions s
LEFT JOIN receipt_items ri ON ri.submission_id = s.submission_id
GROUP BY s.submission_id, s.user_uuid, s.status, s.project_type
HAVING COUNT(ri.item_id) = 3
ORDER BY s.created_at DESC
LIMIT 5;
```

- `receipt_count = 3` 인 행만 나옵니다.
- 결과에서 아무 행이나 하나 고르고, 아래 예시의 `SUBMISSION_ID`, `USER_UUID`, `IMAGE_KEY_1/2/3` 자리에 **그 행의 값**을 넣어 사용하면 됩니다.

### 1-2. 특정 서브미션의 image_key 3개만 따로 보고 싶을 때

```sql
-- 위에서 얻은 submission_id 로 치환
SELECT item_id, submission_id, seq_no, image_key, doc_type
FROM receipt_items
WHERE submission_id = '여기에_submission_id_입력'
ORDER BY seq_no, item_id;
```

- 여기서 나온 `image_key` 3개를 Complete 요청의 `documents[].imageKey` 에 그대로 넣으면 됩니다.

---

## 2. 예시 값 (DB 조회 결과를 넣은 형태)

SQL 결과가 아래와 같다고 가정합니다.

| submission_id | user_uuid   | status   | project_type | receipt_count | image_keys |
|---------------|-------------|----------|--------------|---------------|------------|
| a1b2c3d4-...  | user-abc-1  | PENDING  | TOUR         | 3             | {TOUR/receipts/a1b2..._a1.jpg, TOUR/receipts/a1b2..._a2.jpg, TOUR/receipts/a1b2..._a3.jpg} |

이때:

- `SUBMISSION_ID` = `a1b2c3d4-...` (실제 UUID)
- `USER_UUID` = `user-abc-1`
- `IMAGE_KEY_1` = `TOUR/receipts/a1b2..._a1.jpg`
- `IMAGE_KEY_2` = `TOUR/receipts/a1b2..._a2.jpg`
- `IMAGE_KEY_3` = `TOUR/receipts/a1b2..._a3.jpg`

아래 curl에서 이 이름들을 **자신의 DB 조회 결과**로 바꿔서 사용하면 됩니다.

---

## 3. API 테스트 (curl)

Base URL은 로컬이면 `http://localhost:8000`, 운영이면 `https://api.nanum.online` 등으로 바꿉니다.

### 3-1. Presigned URL — 1장째 (receiptId 없이, 새 서브미션 생성)

```bash
curl -s -X POST "http://localhost:8000/api/v1/receipts/presigned-url?fileName=receipt1.jpg&contentType=image%2Fjpeg&userUuid=USER_UUID&type=TOUR" | jq .
```

- 응답에 `receiptId`, `objectKey` 가 나옵니다. **이 receiptId를 2·3장째와 Complete에서 동일하게 사용**해야 합니다.
- 실제 테스트 시 `USER_UUID` 를 1-1에서 조회한 `user_uuid` 로 바꿉니다. (새로 테스트할 때는 아무 문자열이라도 **동일한 값**을 계속 씁니다.)

### 3-2. Presigned URL — 2장째·3장째 (같은 receiptId + 같은 userUuid)

```bash
# 2장째 (위에서 받은 receiptId 사용)
curl -s -X POST "http://localhost:8000/api/v1/receipts/presigned-url?fileName=receipt2.jpg&contentType=image%2Fjpeg&userUuid=USER_UUID&type=TOUR&receiptId=RECEIPT_ID" | jq .

# 3장째
curl -s -X POST "http://localhost:8000/api/v1/receipts/presigned-url?fileName=receipt3.jpg&contentType=image%2Fjpeg&userUuid=USER_UUID&type=TOUR&receiptId=RECEIPT_ID" | jq .
```

- `USER_UUID`: 1장째와 **동일**
- `RECEIPT_ID`: 1장째 presigned 응답의 `receiptId`
- 이렇게 해야 **같은 서브미션 1건에 영수증 3장**이 묶이고, 403 owner mismatch 가 나지 않습니다.

### 3-3. Complete — 영수증 3장 제출 (DB 기존 서브미션으로 테스트할 때)

**이미 DB에 submission 1건 + receipt_items 3건이 있을 때** 그 submission을 “완료” 처리하려면, 해당 submission의 `submission_id` / `user_uuid` / 3개의 `image_key` 를 1-1·1-2 SQL로 조회한 뒤 아래처럼 넣습니다.

```bash
export BASE="http://localhost:8000"
export SUBMISSION_ID="a1b2c3d4-xxxx-xxxx-xxxx-xxxxxxxxxxxx"   # DB의 submission_id
export USER_UUID="user-abc-1"                                   # DB의 user_uuid
export IMAGE_KEY_1="TOUR/receipts/a1b2c3d4_abc1_receipt1.jpg"  # receipt_items.image_key 1
export IMAGE_KEY_2="TOUR/receipts/a1b2c3d4_abc2_receipt2.jpg"   # receipt_items.image_key 2
export IMAGE_KEY_3="TOUR/receipts/a1b2c3d4_abc3_receipt3.jpg"   # receipt_items.image_key 3

curl -s -X POST "${BASE}/api/v1/receipts/complete" \
  -H "Content-Type: application/json" \
  -d "{
    \"receiptId\": \"${SUBMISSION_ID}\",
    \"userUuid\": \"${USER_UUID}\",
    \"type\": \"TOUR\",
    \"documents\": [
      { \"imageKey\": \"${IMAGE_KEY_1}\", \"docType\": \"RECEIPT\" },
      { \"imageKey\": \"${IMAGE_KEY_2}\", \"docType\": \"RECEIPT\" },
      { \"imageKey\": \"${IMAGE_KEY_3}\", \"docType\": \"RECEIPT\" }
    ],
    \"data\": {
      \"items\": [
        { \"amount\": 50000, \"payDate\": \"2026-03-01\", \"storeName\": \"상점1\", \"location\": \"서울\", \"cardPrefix\": \"1000\" },
        { \"amount\": 30000, \"payDate\": \"2026-03-01\", \"storeName\": \"상점2\", \"location\": \"서울\", \"cardPrefix\": \"1000\" },
        { \"amount\": 20000, \"payDate\": \"2026-03-01\", \"storeName\": \"상점3\", \"location\": \"서울\", \"cardPrefix\": \"1000\" }
      ]
    }
  }" | jq .
```

- `receiptId` = DB의 `submission_id`
- `userUuid` = DB의 `user_uuid` (다르면 403 "receiptId owner mismatch")
- `documents[].imageKey` = 해당 서브미션의 `receipt_items.image_key` 3개
- TOUR는 1~3장 RECEIPT만 허용되므로 위처럼 3개면 됩니다.

## 4. 403 "receiptId owner mismatch" 확인용 (의도적으로 실패시키기)

DB에 있는 **실제 submission_id**와 **다른 user_uuid**를 넣으면 403이 나와야 합니다.

```bash
curl -s -X POST "http://localhost:8000/api/v1/receipts/complete" \
  -H "Content-Type: application/json" \
  -d "{
    \"receiptId\": \"DB에서_조회한_submission_id\",
    \"userUuid\": \"다른_사용자_uuid\",
    \"type\": \"TOUR\",
    \"documents\": [
      { \"imageKey\": \"이미지키1\", \"docType\": \"RECEIPT\" },
      { \"imageKey\": \"이미지키2\", \"docType\": \"RECEIPT\" },
      { \"imageKey\": \"이미지키3\", \"docType\": \"RECEIPT\" }
    ]
  }" | jq .
```

- 기대: `403` + `{"detail":"receiptId owner mismatch"}`

---

## 5. 요약

| 단계 | 사용할 DB 값 | 비고 |
|------|----------------|------|
| Presigned 1장 | `userUuid` (원하는 값으로 고정) | receiptId 없음 → 새 submission 생성 |
| Presigned 2·3장 | 1장에서 받은 `receiptId` + **같은** `userUuid` | 같은 서브미션에 이미지 추가 |
| Complete | `submission_id` → receiptId, `user_uuid` → userUuid, `receipt_items.image_key` 3개 → documents[].imageKey | 세 값이 한 서브미션에서 나온 것과 일치해야 403 없음 |

- **영수증 3장 = receiptId 1개 + userUuid 1개를 presigned/complete 모두에서 동일하게 사용**하면 됩니다.
- DB 기준 예시는 **1. SQL로 submission_id, user_uuid, image_key 3개**를 뽑은 뒤, **3-3 curl**의 변수만 그 값으로 바꿔서 쓰면 됩니다.
