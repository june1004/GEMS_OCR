# GEMS Admin 백엔드(BE) 요청 사항 정리

관리자 페이지(프론트엔드)에서 필요로 하는 API 및 데이터 요청 사항을 정리한 문서입니다.

---

## 1. 로그인 API – 조직 정보 포함

**상세:** [로그인_API_조직정보_요청사항.md](./로그인_API_조직정보_요청사항.md)

- **POST /api/v1/auth/login** 응답의 `user` 객체에 아래 필드 포함 요청.
  - `organization_id` / `organizationId`: 조직 ID (미반환 시 프론트 "api (미반환)" 표시)
  - `org_name` / `orgName`: (선택) 조직명
  - `org_type` / `orgType`: (선택) 조직 유형
  - `campaignIds`: (선택) 해당 사용자가 접근 가능한 캠페인 ID 배열 → 기관/검수자 권한 필터에 사용

## 2. 제출(Submissions) API – 대시보드·검수용

현재 **GET /api/v1/admin/submissions** 를 전제로 한 추가/확인 사항입니다.

### 2.0 422 Unprocessable Content 대응 (우선 요청)

아래와 같이 호출 시 **422**가 발생하고 있습니다.  
→ **원인**: 쿼리 파라미터(`status`, `from`, `to`, `campaignId`)를 BE에서 아직 허용하지 않거나, 파라미터 이름이 다를 가능성.

| FE가 보내는 예시 | 용도 |
|------------------|------|
| `?status=APPROVED&campaignId=1&limit=1000&offset=0` | 승인 건 금액 합계(예산 소진률) |
| `?from=2026-03-04&to=2026-03-10&campaignId=1&limit=500` | 최근 7일 접수 추이·유형별 비중 |

**BE 측 요청 사항**

- **옵션 A**: 위 파라미터 이름 그대로 지원  
  - `status`, `from`, `to`, `campaignId` (camelCase), `limit`, `offset` 수용
- **옵션 B**: snake_case만 지원하는 경우  
  - 지원하는 파라미터 이름을 알려주시면 FE에서 맞춰서 요청하도록 수정 가능  
  - 예: `campaign_id`, `date_from`, `date_to` 등

422가 나오는 동안은 FE에서 해당 요청을 빈 데이터(`total: 0`, `items: []`)로 처리해 두었으며, BE 반영 후에는 자동으로 실제 데이터가 표시됩니다.

---

### 2.1 쿼리 파라미터 지원

| 파라미터 | 용도 | 비고 |
|----------|------|------|
| `from` | 시작일 (YYYY-MM-DD) | 금일/어제/최근 7일 집계에 사용 |
| `to` | 종료일 (YYYY-MM-DD) | |
| `campaignId` | 캠페인 ID로 필터 | 기관·검수자: 자신의 캠페인만 조회 |
| `status` | MANUAL_REVIEW, APPROVED, **FIT** 등 | 검수 대기 건수, 승인/FIT 건 집계에 사용. 대시보드 "STAY/TOUR별 FIT 건수"는 `status=FIT` + `project_type=STAY|TOUR` 조합으로 조회 |
| `project_type` | **STAY**, **TOUR** | 업종별 필터. 대시보드 "전체 건수·FIT 건수 (STAY/TOUR)" 카드에서 업종별 전체·FIT 건수 표시에 사용 |
| `limit`, `offset` | 페이징 | |

- **응답**: `{ total: number, items: AdminSubmissionSummary[] }` 형태 유지 요청.
- **items[]** 각 항목에 **project_type** (또는 projectType) 필드 있으면 좋음: `STAY`(숙박) / `TOUR`(일반 소비) 등. 대시보드·서브미션 업종 필터에 사용.

**제출 항목 확장 필드 (선택, FE에서 미제공 시 비활성 표시)**

| 필드 | 용도 |
|------|------|
| `thumbnail_url` | 서브미션 목록·호버 시 썸네일 미리보기 |
| `confidence` | 신뢰도 슬라이더 필터 (0~100, 낮은 건만 검수 우선) |
| `integrityCheck` (boolean) | 무결성 검사 결과, 목록에 OK 표시 |

### 2.2 (선택) 대시보드 전용 집계 API

한 번의 호출로 대시보드 수치를 내려받고 싶을 경우 아래 형태의 API를 검토해 주시면 좋습니다.

- **GET /api/v1/admin/dashboard/stats** (또는 유사 경로)
- **쿼리**: `campaignId`, `from`, `to` (예: 오늘, 어제, 최근 7일)
- **응답 예시**:
  - `todayCount`, `yesterdayCount`: 금일/전일 제출 건수 (전일 대비 증감률 계산용)
  - `pendingCount`: MANUAL_REVIEW 건수
  - `approvedAmountSum`: 승인(APPROVED) 건의 total_amount 합계 (예산 소진률용)
  - `byCategory`: `{ STAY: number, TOUR: number }` 등 (유형별 건수)
  - `dailyCounts`: `{ date: string, count: number }[]` (일자별 추이 차트용)

미구현 시에도 현재처럼 **GET /api/v1/admin/submissions** 여러 번 호출로 프론트에서 집계 가능하나, 성능·일관성 측면에서 집계 API가 있으면 유리합니다.

- **상태별 집계 UI/UX:** 대시보드의 상태별 건수(적합|부적합|대기 및 검증|에러) 화면에 대한 시각화·보안·AI 자산화 지침은 [상태별_집계_디자인_지침.md](./상태별_집계_디자인_지침.md)를 참고합니다.

### 2.3 대시보드 정확한 표시를 위한 BE 확인·요청 사항

대시보드가 **GET /api/v1/admin/submissions** 결과만으로 정확히 표시되려면 아래를 BE에서 확인·적용해 주시면 됩니다.

| 구분 | FE 동작 | BE 요청 사항 |
|------|---------|----------------|
| **일자별 접수 추이** | `from`/`to`에 YYYY-MM-DD 전달, **created_at** 기준 일별 건수 사용 | 날짜 필터는 **submissions.created_at** 기준으로만 적용하면 됨. (선택) **updated_at** 기준 옵션을 쓰려면 쿼리 파라미터 `dateField=created_at` / `dateField=updated_at` 지원. 타임존: DB가 UTC일 때 클라이언트 "오늘"과 하루 어긋날 수 있으므로, 필요 시 `timezone=Asia/Seoul` 등으로 **해당 TZ의 날짜**로 집계해 주면 좋음. |
| **검수 대기(대기중)** | `statusStage=MANUAL_REVIEW`, `limit=1` 로 조회 후 **total** 사용 | **status_stage** 컬럼을 검수 대기 여부에 맞게 채워 주기. (검토 대기 = `MANUAL_REVIEW`, 자동 처리 완료 = `DONE` 등) |
| **총 제출건수 / 금일 제출건수** | 필터 없음·날짜만 주어서 조회, **total** 사용 | 동일 API의 **total**이 조건에 맞는 전체 건수임을 보장. (receipt_items 존재 여부 등 기존 조건 유지) |
| **승인 금액 합계(예산 소진률)** | `status=APPROVED`, `limit=1000` 요청 후 items의 **total_amount** 합산 | 현재 BE **limit 상한이 200**이라 최대 200건만 합산됨. → **(권장)** `status=APPROVED` 인 건의 **SUM(total_amount)** 만 반환하는 전용 파라미터(예: `aggregate=sum`) 또는 **GET /api/v1/admin/dashboard/stats** 등 집계 전용 API에서 **approvedAmountSum** 제공. 또는 집계 전용으로 limit 상한 완화(예: 5000). |
| **최근 접수 5건** | `limit=5`, **items[]** 에서 receiptId, project_type, status, total_amount 사용 | 응답 **items[]** 에 **project_type**(STAY/TOUR), **status**, **total_amount**, **created_at** 포함되어 있으면 됨. (이미 스펙에 있으면 유지) |

**정리**

- **반드시**: `statusStage` 필터 지원, 날짜 필터는 **created_at** 기준 당일 전체(00:00 ~ 다음날 00:00 미만) 적용, **total** 정확 반환.
- **권장**: 승인 금액 합계는 **aggregate=sum** 또는 **dashboard/stats** 등으로 **전체 승인 건 합계** 제공.
- **선택**: 일자별 추이용 **timezone** 파라미터, **updated_at** 기준 옵션, 대시보드 일괄 집계 API(**dashboard/stats**).

**대시보드 네트워크 동작 (참고)**

- FE는 금일 신규 접수·일자별 접수 추이를 위해 **같은 GET /admin/submissions**를 여러 번 호출합니다.
- **금일 신규 접수**: `from=오늘&to=오늘&dateFrom=오늘&dateTo=오늘&campaignId=1&limit=1` → 응답의 **total**만 사용합니다.
- **일자별 접수 추이**: 최근 7일 각각에 대해 `from=해당일&to=해당일&...&limit=1` 호출 → 각 응답의 **total**을 해당 일자의 건수로 사용합니다.
- `limit=1`은 전송량 절감용이며, **total은 “해당 조건에 맞는 전체 건수”**여야 합니다. (items 길이가 아니라 count(\*) 결과를 total로 반환해야 함.)  
  → BE가 total을 정확히 반환하면 금일/일자별 수치가 올바르게 표시됩니다.
- **BE가 from/to를 미지원하는 경우**: FE는 `from`/`to` 없이 최근 500건을 조회한 뒤, 각 항목의 **created_at**으로 일자별 건수를 클라이언트에서 집계해 차트에 표시합니다. 이 경우 500건을 초과하는 구간은 집계에 포함되지 않으므로, 정확한 일자별 추이를 위해서는 **from/to(또는 date_from/date_to) 지원 및 created_at 기준 일별 total 반환**을 권장합니다.

## 3. 일괄 반려 API (Bulk Rejection)

서브미션(제출 검색) 페이지에서 **일괄 반려 사유 입력 팝업**으로 선택 건에 대해 반려 처리·알림·학습 데이터 태깅을 할 수 있도록, 아래 API 및 연동을 요청합니다.

### 3.1 일괄 반려 실행 API

- **POST /api/v1/admin/submissions/bulk-reject** (또는 유사 경로)
- **Request Body 예시**:
  ```json
  {
    "receiptIds": ["uuid-1", "uuid-2", ...],
    "reasonCode": "image_unreadable",
    "reasonMessage": "제출하신 영수증의 글자가 흐려 금액 확인이 어렵습니다. 재촬영 후 제출 바랍니다.",
    "tagAsError": true
  }
  ```
  - `reasonCode`: 프리셋 ID (예: image_unreadable, duplicate, out_of_scope, below_min_amount)
  - `reasonMessage`: 시민에게 전달할 반려 사유 문구 (알림톡 등에 사용)
  - `tagAsError`: true 시 Sidecar JSON 등에 `is_error: true` 학습 데이터 태그

- **기대 동작**:
  - 각 submission의 `status`를 `UNFIT` 또는 `REJECTED`로 변경
  - (선택) 각 userUuid에 매핑된 연락처로 반려 사유 포함 카카오 알림톡/푸시 발송
  - (선택) 보안 스토리지 Sidecar JSON에 `rejection_reason`, 검수자 ID, 일시 기록
  - `tagAsError: true`인 경우 해당 건을 부정수급/오류 학습 데이터셋으로 분류

### 3.2 감사(Audit) 로그

- 일괄 반려 실행 시 **누가, 언제, 몇 건, 어떤 사유로** 실행했는지 Audit Log에 기록 요청 (IP 포함 시 지자체 보안 감사 대비).

## 4. 반려 사유 집계 (선택)

대시보드 “주요 반려 사유 Top 3” 표시를 위해, 반려 사유별 건수를 내려주는 API가 있으면 좋습니다.

- 예: **GET /api/v1/admin/receipts/reject-reasons** (또는 기존 제출/영수증 API 확장)
- **응답 예시**: `[{ reason: "이미지 흐림", count: 32 }, { reason: "금액 불일치", count: 15 }, ...]`
- 미구현 시 프론트에서는 “API 연동 시 표시” 문구만 노출합니다.

## 5. 운영 컨텍스트 (Admin Context)

- **GET /api/v1/admin/context** (또는 **GET /api/v1/admin/me**): 현재 담당자 정보와 함께 **projectId**, **lastSelectedCampaignId**, **lastSelectedProjectId** 반환. FE는 캠페인 선택 시 GET을 다시 호출하고, 응답의 projectId가 선택한 캠페인과 같을 때만 그 값으로 폼을 채움.
- **PUT /api/v1/admin/context**: Request Body에 **projectId**, **lastSelectedCampaignId**, **lastSelectedProjectId** 중 전송한 값을 **사용자당 1건**만 저장. 저장 후 동일한 MeResponse를 반환.

**저장 구조 (이 레포 BE)**

- **마지막에 저장한 1건만 유지**: 별도 API 없이 GET/PUT만으로 동작. `admin_users.last_selected_campaign_id`, `last_selected_project_id` 컬럼에 담당자별로 1건만 갱신. 캠페인 B를 저장하면 이전에 저장한 캠페인 A의 선택값은 덮어써지며, 나중에 A를 다시 선택해도 "A용으로 저장했던 운영 컨텍스트"는 복원되지 않고, A의 캠페인(사업) 상세만 보이게 됨.
- **캠페인별로 각각 유지**하려면: (user, projectId) 단위 저장·조회가 필요. 예: **GET /api/v1/admin/context?projectId=xxx** 로 해당 캠페인 저장값만 내려주거나, GET 응답에 캠페인별 맵/목록을 넣는 방식으로 확장 필요.

**변경 이력 (감사 연동)**

- 캠페인 선택(관리 주체·기관명) 저장 시 BE가 **감사 로그**에 `CONTEXT_UPDATE` 액션으로 기록. **GET /api/v1/admin/context/history** 로 현재 담당자의 변경 이력만 조회 가능.
- FE "변경 이력 보기" 클릭 시: 위 **GET /api/v1/admin/context/history** 호출 후 `items[]`(created_at, actor, projectId, lastSelectedCampaignId, before_*) 를 표시하면 "관리 주체·기관명 변경 이력은 BE 감사 로그 연동 시 표시됩니다" 토스트를 실제 목록으로 대체할 수 있음.

## 6. 캠페인(Campaigns) API

- **GET /api/v1/admin/campaigns**, **GET /api/v1/admin/campaigns/:id**  
  - 응답에 **start_date**, **end_date**, **budget** 포함되어 있으면 대시보드 “캠페인 기간(D-Day)”, “예산 소진 현황”에 사용합니다.
- **campaignId** 기준 제출/집계 API와 연동되면, 기관·검수자는 자신에게 허용된 캠페인만 조회 가능해야 합니다.

## 7. 영수증 이미지 (보안 스토리지)

- **GET /api/v1/admin/receipts/:receiptId/images**  
  - 보안 스토리지(Secure Storage) 사용 시, 원본 이미지는 **임시 Signed URL** 등으로 단기간만 노출되도록 구성해 주시면, 지자체 보안 가이드라인 준수에 유리합니다.
  - **인증 필수**: 이 엔드포인트는 **X-Admin-Key** 또는 **Bearer 토큰**이 없으면 **401 Unauthorized** (`Authorization required (X-Admin-Key or Bearer token)`)를 반환합니다. Swagger/Postman 등으로 테스트할 때는 반드시 인증 헤더를 넣어야 합니다.
  - **이미지가 없는 영수증**: 해당 receiptId에 이미지가 없어도 **401을 반환하면 안 됩니다.** 인증이 성공했을 때는 **200 OK + `{ receiptId, items: [] }`** 형태로 응답해 주세요. 401은 **인증 실패 시에만** 사용하고, "이미지 없음"은 빈 `items` 배열로 구분합니다.

### 7.1 net::ERR_BLOCKED_BY_ORB (이미지 로딩 실패)

**증상:** 개발자 도구 Network 탭에서 이미지 요청이 `(failed) net::ERR_BLOCKED_BY_ORB` 로 막히고, 영수증 썸네일이 빨간 X로 표시됩니다.

**원인:** 브라우저의 **Cross-Origin Read Blocking(ORB)** 정책입니다. FE(예: localhost 또는 GEMS 도메인)가 **다른 오리진**(API 서버, S3/MinIO 등)의 이미지 URL을 `<img src="...">` 로 요청할 때, 서버가 내려주는 응답의 **Content-Type**이 이미지로 인식되지 않거나, 스니핑 방지 정책에 걸리면 차단됩니다.  
예: Signed URL로 이미지를 내려주지만 응답 헤더가 `Content-Type: application/octet-stream` 이거나 없음, 또는 실제 바디가 선언된 타입과 다름.

**해결 (BE/스토리지 측):**

| 조치 | 설명 |
|------|------|
| **1) 올바른 Content-Type** | 이미지 URL(Signed URL 포함)을 제공하는 서버·스토리지가 **실제 파일 형식에 맞게** `Content-Type: image/jpeg`, `image/png` 등을 반환하도록 설정합니다. S3/MinIO 객체 업로드 시 메타데이터에 `Content-Type`을 넣거나, API 게이트웨이에서 프록시 시 `Content-Type`을 덮어쓰세요. |
| **2) X-Content-Type-Options** | `X-Content-Type-Options: nosniff` 를 붙이면 브라우저가 선언된 타입을 신뢰해 ORB 차단 가능성이 줄어듭니다. |
| **3) 같은 오리진으로 프록시 (권장)** | FE와 **같은 오리진**에서 이미지를 내려주면 ORB가 걸리지 않습니다. 예: BE에서 **GET /api/v1/admin/receipts/:receiptId/images/:itemId/bytes** 처럼 이미지 바이트를 가져와서 `Content-Type: image/jpeg` 등으로 그대로 스트리밍해 주는 프록시 엔드포인트를 두고, FE는 `image_url`로 이 프록시 URL만 사용하도록 하면 됩니다. (Signed URL을 FE에 노출하지 않고, BE가 스토리지에서 읽어서 같은 오리진으로 응답) |

**FE 동작:** 이미지 로드는 `<img src={image_url}>` 이며, `image_url`이 BE에서 내려준 값 그대로 사용됩니다. BE가 위와 같이 Content-Type을 맞추거나 같은 오리진 프록시를 제공하면 ORB 오류가 사라집니다.

## 10. 증거(영수증) 확인 – 데이터 교정 및 Sidecar (Data Curating)

증거 확인 단계는 단순 조회가 아니라 **데이터 교정(Data Curating)**의 핵심입니다. 담당자가 실패 사유(금액 미달, 지역 불일치 등)에 맞춰 값을 수정하고, **수정 사유**와 함께 **Sidecar JSON**에 기록되어야 AI 자산화·재학습 정답지(Ground Truth)로 활용할 수 있습니다.

**현재 FE 동작:** "교정 사유 및 값 저장" 클릭 시 **로컬(localStorage)**에만 저장됩니다. Sidecar JSON·AI 학습 반영을 위해서는 **BE 교정 API 구현이 필요**합니다.

**→ BE 연동 상세 지침:** [BE_교정_API_연동_지침.md](./BE_교정_API_연동_지침.md) (엔드포인트, Request Body, Sidecar 구조, 체크리스트)

### 10.1 교정 데이터 반영 API (권장)

- **PATCH /api/v1/admin/submissions/:receiptId/correction** (또는 PUT /receipts/:receiptId/sidecar 등)
- **Request Body 예시**:
  ```json
  {
    "total_amount": 65000,
    "address": "강원도 춘천시 ...",
    "correction_reason_code": "ocr_error",
    "correction_reason_detail": "빛 반사로 금액 인식 오류",
    "audit": { "actor_id": "admin@example.com", "at": "2026-03-05T12:00:00Z" }
  }
  ```
- **기대 동작**:
  - 해당 submission의 `total_amount` 등 DB 컬럼 업데이트
  - **Sidecar JSON**에 `[기존 값] → [수정 값] (수정자 ID, 시각)` 형태의 교정 이력 추가
  - `correction_reason_code`: `ocr_error` | `user_mistake` | `amount_sum_error` | `policy_exception` | `address_mismatch` | `other`
  - 교정 이력이 있으면 향후 AI 재학습 시 "사용자 입력이 맞고 OCR이 틀린 케이스" 등으로 분류해 학습 데이터로 활용

### 10.2 Sidecar JSON 구조 (참고)

보안 스토리지에 저장되는 Sidecar에는 다음이 포함되면 좋습니다.

- `receipt_id`: 제출(영수증) ID
- `ai_result`: OCR/AI 추출 결과 (amount, confidence 등)
- `human_correction`: 담당자 교정 정보
  - `final_amount`, `final_address` 등 확정값
  - `reason_code`: [GEMS 표준 수정·반려 사유 분류](./GEMS_표준_수정_반려_사유_분류.md)의 코드 (예: `ERR_OCR_AMOUNT`, `USER_AMOUNT_MISTAKE`)
  - `reason_desc`: 상세 설명 (또는 프리셋 detail)
  - `reviewed_by`, `at`
- `asset_tag`: 학습 데이터 분류용 (예: `RE_TRAINING_REQUIRED`, `USER_ERROR_LABEL`, `LOW_QUALITY_SAMPLE`, `FRAUD_CHECK`)

**표준 reason_code·asset_tag**는 [GEMS_표준_수정_반려_사유_분류.md](./GEMS_표준_수정_반려_사유_분류.md)를 따르면, AI 재학습 시 "왜 불일치했는가" 라벨로 활용할 수 있습니다.

### 10.3 증거 확인 – 판정·금액·날짜·카드번호 BE 지침

FE는 API 응답을 그대로 표시합니다. 아래는 **판정·데이터 일관성**을 위한 BE/API 요청 사항입니다.

| 구분 | 요청 사항 |
|------|-----------|
| **판정 (FIT)** | **최대 3장 입력 영수증 중 한 장이라도 정상조건을 만족하면** 제출 건 전체를 **FIT**으로 판정·반영해야 합니다. 실패 사유가 "합산 금액 미달"로 나오는데 장별 금액·합산이 기준을 충족하는 경우, 이는 표시 문제가 아니라 **판정 로직**이 한 장이라도 정상이면 FIT을 반환하도록 수정되어야 합니다. (overall_status, status, total_amount 등 API가 FIT과 기준 충족 금액을 반영) |
| **API 금액** | **2. OCR 인식** 등 FE에 노출되는 금액은 **API가 반영한 값**을 사용합니다. 장별 합산이 있는데 제출 단위 `total_amount`가 0이거나 비어 있으면 FE에는 0원으로 표시됩니다. BE에서 장별(item) 금액을 합산해 제출 단위 `total_amount`에 반영하거나, 상태 API가 금액을 정확히 내려주어야 합니다. **STAY인 경우** 합산 시 **첫 번째 장(영수증) 금액만** 사용 ([§10.6](./백엔드_요청사항_정리.md#106-stay-업종-영수증-1매--거래명세서-1부--금액은-첫-번째-이미지만)). |
| **금액 부호** | **금액은 저장·API 모두 양수**로 통일합니다. 앱에서 "지출"이라 **-122,200원**처럼 보여주는 것은 표시 규칙일 뿐, API/DB에는 **122200**처럼 양수로 내려주고 저장합니다. ([BE_교정_API_연동_지침 §2.3](./BE_교정_API_연동_지침.md#23-금액-앞-마이너스－-처리)) |
| **UNFIT_DATE** | **실제 날짜가 맞다면** 교정 저장의 문제가 아니라 **정상으로 처리**되어야 하며, **FIT 판정에 영향을 주면 안 됩니다.** BE 판정 로직에서 날짜 형식/오인식만으로 UNFIT_DATE를 적용한 경우, 실제 날짜가 유효하면 해당 건을 FIT으로 처리하거나 UNFIT_DATE가 최종 FIT을 막지 않도록 수정이 필요합니다. |
| **카드번호 (앞 4자리)** | **card_num(카드번호)은 앞 4자리를 저장·반영**해야 합니다. FE는 4자리일 경우 "앞 4자리"로 해석해 `1234-****-****-****` 형태로 표시합니다. **BE/OCR 저장 로직 지침:** 4자리만 추출될 때는 **앞 4자리**로 해석해 DB/API에 저장하고, 뒷 4자리만 있는 경우에는 저장하지 않거나 별도 필드로 구분하는 것을 권장합니다. 영수증 이미지에서 카드번호 마스킹 시 앞 4자리가 노출되는 경우가 많으므로, OCR 결과 또는 수동 입력값이 4자리일 때는 **앞 4자리**로 저장하도록 규칙을 적용해 주세요. |

### 10.4 상세 내역: 거래명(상호) vs 사용처 불일치 처리

**상황:** 상세 내역 화면에서 **거래명(예: 세인트존스호텔)**과 **사용처(예: 엘케이매니지먼트)**가 서로 다르게 보이는 경우.

| 구분 | 설명 |
|------|------|
| **원인** | 거래명은 **영수증 OCR 상호**(또는 브랜드명), 사용처는 **마스터/사업자 등록 상호**(운영사·법인명)로 서로 다른 데이터 소스일 수 있음. 호텔 이름 ≠ 사업자 상호는 흔한 케이스. |
| **정상 케이스** | 브랜드명(세인트존스호텔)과 운영사(엘케이매니지먼트)가 다른 것은 **그대로 두면 됨**. FE/앱에서는 거래명은 OCR/영수증 기준, 사용처는 마스터 매칭 결과로 표시하고, 사용자에게 "사용처는 등록된 사업자 상호입니다" 정도 안내만 해도 됨. |
| **검수 시** | 관리자 **증거 확인**에서 OCR 상호(`store_name`)와 사용처(마스터 매칭 결과)를 둘 다 노출해, **잘못 매칭**된 경우에만 교정(주소·상호 수정, 신규 상점 승인 등)으로 조치. PENDING_NEW면 신규 상점 정책(자동/수동)에 따라 처리. |
| **BE 권장** | (1) **거래명/표시명**: OCR 추출 `store_name` 또는 앱에서 쓰는 제목용 필드. (2) **사용처**: 마스터/매칭 결과 상호(`merchant_name`, `registered_store_name` 등). API에서 두 필드를 구분해 내려주면 FE/앱에서 "거래명 ≠ 사용처"인 이유를 설명하기 쉽고, 교정 시 어떤 값을 수정할지 명확해짐. |

### 10.5 목록 금액: 최종 확정(교정) 금액 반영

**요구:** 제출/결과 **목록**의 금액 컬럼은 **수정 없으면 OCR 인식값**, **검수자가 최종 확정으로 수정했으면 수정한 금액**이 표시되어야 합니다.

| 구분 | 설명 |
|------|------|
| **FE 동작** | `GET /api/v1/admin/submissions` 응답의 각 항목에서 **`final_amount`**(또는 `corrected_total_amount`, `correctedTotalAmount`)가 있으면 목록·정렬·집계에 그 값을 사용하고, 없으면 `total_amount`(OCR 값)를 사용합니다. |
| **BE 권장** | (1) **방법 A**: 교정 저장 시 `submissions.total_amount`를 최종 확정 금액으로 업데이트하고, 목록 API는 기존처럼 `total_amount`만 반환. (2) **방법 B**: `total_amount`는 OCR 값으로 유지하고, 교정 시 Sidecar/별도 컬럼에만 저장한 뒤, 목록 API 응답에 **`final_amount`**(또는 `corrected_total_amount`)를 포함해 주기. 교정 이력이 있으면 해당 값을, 없으면 null. FE는 `final_amount ?? total_amount`로 표시합니다. |

### 10.6 STAY 업종: 영수증 1매 + 거래명세서 1부 — 금액은 첫 번째 이미지만

**업종 구조**

- **STAY(숙박)** 제출 건 = **영수증 1매** + **거래명세서 1부** (이미지 2장).
- **첫 번째 이미지(영수증)**: 결제 금액·일자 등 **금액 및 결제 정보**는 **이 이미지에서만** 수집·저장합니다.
- **두 번째 이미지(거래명세서)**: **실제 사용처** 정보(상호명, 주소 등) 수집용. 거래명세서에도 금액이 있으나, **거래명세서 템플릿 고도화 전까지는 거래명세서 내 금액은 무시**하고, **영수증(첫 번째 이미지) 금액만** 제출 건 금액으로 사용합니다.

**합산 오류 방지**

- 현재 **합산에서 과도한 금액**이 나오는 이유: 영수증 금액과 거래명세서 금액을 둘 다 더해 버리는 경우.
- **해결:** **project_type = STAY** 인 제출 건에 대해서는 **`total_amount`(및 판정/재처리 시 사용하는 합산값)를 “첫 번째 receipt_item의 amount만”** 사용해야 합니다. 두 번째 이후 항목(거래명세서)의 amount는 **합산·판정·API 응답에 포함하지 않습니다.**

**BE/API 수정 사항 (지침)**

| 구분 | 수정 내용 |
|------|-----------|
| **금액 집계 (신규 저장)** | STAY인 경우: `receipt_items`를 **정렬 순서(created_at 또는 order)** 기준으로 **첫 번째 항목만** 보고, 해당 항목의 `amount`(또는 `extracted_data.amount`)만 `submissions.total_amount`에 반영. 두 번째·세 번째 항목의 금액은 **합산하지 않음**. |
| **금액 집계 (재처리)** | 재처리 시 `total_amount` 재계산할 때도 **project_type = STAY**이면 **첫 번째 receipt_item의 amount만** 사용. ([BE_교정_API_연동_지침 §6.2](./BE_교정_API_연동_지침.md#62-재처리-api-로직정책만-재적용-ocr-미호출) 재처리 로직 참고) |
| **재처리 콜백** | 재처리 후 **FIT으로 변경된 건**(previous_status !== FIT, new_status === FIT)에만 사용자에게 콜백 URL 자동 전송. **이미 FIT이었던 건**은 콜백 재전송하지 않음. ([BE_교정_API_연동_지침 §6.2](./BE_교정_API_연동_지침.md#62-재처리-api-로직정책만-재적용-ocr-미호출)) |
| **OCR/콜백 파이프라인** | BE가 외부 OCR·콜백에서 `total_amount`를 받아 저장하는 구조라면: **STAY인 경우** 콜백에서 “첫 번째 장만의 금액”을 보내도록 하거나, BE에서 `receipt_items` 기준으로 **첫 번째 항목 금액만** 사용해 `submissions.total_amount`를 덮어쓰도록 처리. |
| **TOUR(일반소비)** | TOUR 등 다른 업종은 기존처럼 **전체 항목 금액 합산** 또는 해당 업종 규칙을 유지해도 됨. (STAY에만 “첫 장만” 규칙 적용) |

**정리**

- **금액**: STAY → **첫 번째 이미지(영수증)에서만** 수집·저장·합산.
- **상호·주소 등**: 거래명세서(두 번째 이미지)에서 수집해 사용처·상호명·주소 등으로 활용.
- 거래명세서 템플릿 고도화 후에는 별도 규칙으로 거래명세서 금액을 반영할 수 있음.

## 13. CORS 및 로컬 개발 (api.nanum.online 연동 시)

**증상:** 프론트를 `http://localhost:8080`에서 띄우고 백엔드 `https://api.nanum.online`으로 요청 시  
`No 'Access-Control-Allow-Origin' header` + **500 Internal Server Error** 가 나올 수 있습니다.

- **500 원인:** 서버 내부 오류(예: DB 컬럼 누락, 예외 미처리). 이 레포의 BE는 submissions 목록에서 `receipt_items.image_url` 없을 때 쿼리 폴백으로 500을 방지하도록 수정되어 있습니다. 배포 시 동일 코드/마이그레이션 적용 필요.
- **CORS:** 500 응답 시 서버가 CORS 헤더를 붙이지 않으면 브라우저가 CORS 오류로 표시합니다. BE `main.py`는 기본 `CORS_ORIGINS`에 `http://localhost:8080`을 넣고, 전역 예외 핸들러로 5xx도 JSONResponse로 반환해 미들웨어(CORS)가 적용되도록 되어 있습니다.
- **배포 서버(api.nanum.online)에서 로컬 프론트 허용:**  
  환경 변수 `CORS_ORIGINS=http://localhost:8080` (또는 `https://localhost:8080` 등 필요 origin을 쉼표로 구분) 설정 시 로컬에서 해당 백엔드로 요청 가능합니다.

### 13.1 관리자 페이지 JWT 로그인 후 401 점검

로그인까지 성공했는데 관리자 API 호출 시 **401 Unauthorized**가 나오면 아래를 BE에서 확인합니다.

| 확인 항목 | BE 동작 (이 레포 기준) | 점검 방법 |
|-----------|------------------------|-----------|
| **JWT 검증** | `Authorization: Bearer <token>` 헤더를 읽어, `Bearer ` 접두어 제거 후 토큰만 추출. `jwt.decode(token, JWT_SECRET, algorithms=["HS256"])`로 **서명 검증** 및 **exp(만료시간) 검사** 수행. 실패 시 401 `Invalid or expired token`. | FE가 요청 시 `Authorization: Bearer <access_token>` 을 정확히 보내는지, BE 로그에 401 원인(만료/서명 불일치)이 남는지 확인. |
| **토큰 만료 시간** | 로그인 시 `exp` = 현재 시각 + **JWT_EXPIRE_MINUTES** (환경 변수, 기본 **480분 = 8시간**). 1시간 이상 권장이므로 기본값 유지 또는 `JWT_EXPIRE_MINUTES=60` 이상 설정. | `JWT_EXPIRE_MINUTES`가 너무 짧지 않은지 확인. (예: 60 이상 권장) |
| **CORS** | 관리자 FE 오리진이 CORS 허용 목록에 있어야 브라우저가 인증 헤더를 실서버로 보낸 뒤 응답을 받을 수 있음. 이 레포 기본값에 `https://gems.nanum.online` 포함. 추가 도메인은 환경 변수 **`CORS_ORIGINS`** 에 쉼표로 나열 (설정 시 목록 전체를 대체). | `https://gems.nanum.online` 및 실제 사용 중인 관리자 페이지 도메인이 `CORS_ORIGINS` 또는 코드 기본 `_cors_list`에 포함되는지 확인. |

- **BE 구현 요약:** `get_admin_context`에서 (1) `X-Admin-Key` 일치 시 슈퍼 관리자, (2) `Authorization: Bearer <token>` 이 있으면 JWT 디코드 → `sub`(이메일), `user_id`, `role`로 DB 사용자 조회 후 캠페인 스코프 적용. JWT_SECRET 미설정 시 로그인/토큰 검증 불가(503).
