# 관리자 페이지 추가 개발 지침

> 기존 `ADMIN_PORTAL_GUIDE.md`에 더해, **추가로 반영된** 관리자 관련 기능·API·설정을 정리한 문서입니다.  
> 관리자 페이지 **추가 개발 시** 이 지침을 기준으로 연동하면 됩니다.

---

## 1. 판정 규칙 — 유효일 설정 (MinIO–DB 정합)

### 1.1 추가된 필드

| 필드 | 타입 | 기본값 | 범위 | 설명 |
|------|------|--------|------|------|
| `orphan_object_days` | int | 1 | 1~365 | **고아 객체 유효일(일)**. MinIO에만 있고 DB에 submission이 없는 객체를 이 일수 경과 후 삭제 후보로 볼 때 사용. |
| `expired_candidate_days` | int | 1 | 1~365 | **만료 후보 유효일(일)**. DB에 submission은 있으나 receipt_items가 없는(Complete 미호출) 건을 만료 후보로 볼 때 사용. |

### 1.2 API

- **조회**: `GET /api/v1/admin/rules/judgment`  
  - 응답에 `orphan_object_days`, `expired_candidate_days` 포함 (없으면 1).
- **수정**: `PUT /api/v1/admin/rules/judgment`  
  - body에 `orphan_object_days`, `expired_candidate_days` 포함 가능. 1~365만 유효.

### 1.3 관리자 화면 반영

- 판정 규칙 설정 화면에 **고아 객체 유효일(일)**, **만료 후보 유효일(일)** 입력 필드 추가.
- 기본값 1일, 1~365 범위 제한.
- 저장 시 기존 판정 규칙과 함께 `PUT /api/v1/admin/rules/judgment`로 전송.

### 1.4 DB

- `judgment_rule_config` 테이블에 컬럼 추가 필요.  
  - 마이그레이션: `PROJECT/migrations/judgment_rule_config_minio_days.sql`

## 2. 신청(submission) 상세 — FE 제출 데이터 스냅샷

### 2.1 추가된 응답 필드

- **API**: `GET /api/v1/admin/submissions/{receiptId}`  
  - 응답에 **`user_input_snapshot`** 포함 (있을 경우).
  - Complete 요청 시 FE가 보낸 `data`(방식2: `data.items[]`)를 그대로 저장한 JSON.  
  - 관리자 화면에서 **FE 입력 vs OCR 결과** 나란히 비교할 때 사용.

### 2.2 관리자 화면 반영

- 신청 상세 화면에서 **FE 제출 데이터(user_input_snapshot)** 를 노출.
  - 예: 장별 사용자 입력 금액·결제일·상호 등과 OCR 결과·이미지를 나란히 표시.

## 3. 이미지 presigned URL — Content-Type 지정

### 3.1 변경 사항

- **API**: `GET /api/v1/admin/receipts/{receiptId}/images`  
  - 응답의 `items[].image_url`은 MinIO presigned GET URL.
  - BE에서 **확장자에 따라** `ResponseContentType` 지정 (`.png` → `image/png`, `.jpg`/`.jpeg` → `image/jpeg`).
  - 저장 시 Content-Type이 잘못돼 있어도 브라우저에서 이미지로 표시되도록 함.

### 3.2 관리자 화면

- 별도 변경 없음. 기존처럼 `image_url`로 이미지 표시하면 됨.

---

## 4. 수동 판정(Override) · 콜백 재전송

- 기존과 동일.  
  - **Override**: `POST /api/v1/admin/submissions/{receiptId}/override` (status, reason, override_reward_amount, resend_callback).  
  - **콜백 재전송**: `POST /api/v1/admin/submissions/{receiptId}/callback/resend` (optional target_url).  
- **콜백 검증(신규, 관리자 클릭 시 즉시 송출)**: `POST /api/v1/admin/submissions/{receiptId}/callback/verify`  
  - BE가 `OCR_RESULT_CALLBACK_URL`로 실제 POST를 보내고, 응답(status/ok/elapsed_ms)을 JSON으로 반환.
- **콜백 전송 로그 조회(신규)**: `GET /api/v1/admin/submissions/{receiptId}/callback/logs` (limit=1~200)  
  - `CALLBACK_SEND`(자동/검증/재전송 실제 송출 결과: status/ok/elapsed_ms/에러), `CALLBACK_VERIFY`(검증 버튼 클릭 기록), `CALLBACK_RESEND`(재전송 트리거 기록) 확인용.
- 상세는 `ADMIN_PORTAL_GUIDE.md` 참고.

## 5. 멈춘 신청(PROCESSING/VERIFYING) 복구

### 5.1 개요

- OCR 분석이 오래 걸리거나 중단되어 **PROCESSING/VERIFYING** 상태에 머무는 건이 있을 수 있음.
- 관리자 페이지에서 **직접 상태를 바꾸는 API는 없음**.  
  - 운영·개발 환경에서 **스크립트** 또는 **SQL**로 복구.

### 5.2 관리자 페이지에서 할 수 있는 것

- **신청 목록/상세**에서 status가 PROCESSING/VERIFYING인 건을 **필터·표시**.
- 안내 문구 예: “일정 시간 이상 처리 중이면 운영자에게 문의. 복구는 백엔드 스크립트로 진행.”

### 5.3 복구 방법 (참고)

- **스크립트**: `PROJECT/scripts/mark_stuck_submissions_error.py` (특정 receiptId 또는 `--all --minutes N`).
- **SQL**: `PROJECT/scripts/mark_stuck_submissions_error.sql` (주석 해제 후 실행).  
- 자세한 사용법: `PROJECT/scripts/README.md`.

## 6. MinIO–DB 정합성 (리콘실리레이션)

### 6.1 개요

- MinIO에는 이미지가 있는데 DB에는 해당 submission이 없거나, Complete가 호출되지 않은 경우가 있음.
- **유효일**은 위 **1. 판정 규칙**의 `orphan_object_days`, `expired_candidate_days`로 관리자가 설정.

### 6.2 관리자 페이지에서 할 수 있는 것

- 유효일 설정은 **판정 규칙 화면**에서만 하면 됨 (1번 참고).
- **리콘실 실행**은 백엔드/배치에서 스크립트로 수행 (`PROJECT/scripts/reconcile_minio_db.py`).  
  - 관리자 페이지에서 “리콘실 실행” 버튼을 둘 경우, 별도 **관리자 전용 API**가 필요하면 BE에 요청.

### 6.3 정책

- 고아 객체: MinIO에만 있음 → 객체 **LastModified** 기준 `orphan_object_days` 일 경과 후 삭제 후보.
- 만료 후보: DB에 submission만 있고 receipt_items 없음 → **created_at** 기준 `expired_candidate_days` 일 경과 후 만료 후보로 보고.  
- 자세한 정책: `MinIO-DB_정합성_및_OCR미인식_정책.md`.

## 8. 카드번호(card_num) 코드 구분

| 값 | 의미 | DB 저장 |
|------|------|--------|
| **0000** | 현금 결제 | 현금일 때만 |
| **1000** | 카드번호 없음 / **** 마스킹 / 미표시 | 카드 정보 없을 때 |
| **4자리 숫자** | 카드번호 마지막 4자리 | OCR에서 인식된 경우 |

- Status·상세 API의 `items[].extracted_data.card_num`(또는 `card_num`)에 위 값이 내려옵니다.
- 관리자·FE에서 노출 시: `0000` → "현금", `1000` → "카드번호 없음", 그 외 → "****-****-****-1234" 등으로 표시하면 됩니다.

## 9. OCR_004 (인식 불량·수동 검수 보정)

- **error_code = OCR_004**: 상점명·사업자번호·주소 등 핵심 필드가 비어 있거나, OCR 컨피던스가 낮을 때 자동 부여.
- 해당 장은 **PENDING_VERIFICATION** 상태로, 관리자 페이지에서 **상세 조회 후 보정**할 수 있도록 노출 권장.
- 보정 후 기존처럼 override API로 FIT/UNFIT 등 최종 판정 처리.
- 환경 변수: `OCR_LOW_CONFIDENCE_REVIEW_THRESHOLD`(기본 70), `OCR_KEY_FIELDS_MIN_FILLED`(기본 2). 자세한 조건은 `네이버_CLOVA_OCR_레퍼런스_및_인식률_검토.md` §6 참고.
